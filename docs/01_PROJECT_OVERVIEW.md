# สรุปภาพรวมโปรเจกต์ MediQueue

## 1. ความหมายและวัตถุประสงค์

**MediQueue** เป็นระบบจัดการคิวโรงพยาบาล (Hospital Queue Management System) ที่ออกแบบมาเพื่อให้ผู้ป่วยจองคิวนัดหมอล่วงหน้า แอดมิน/เจ้าหน้าที่จัดการคิวและสถานะการรับบริการ และแพทย์ดูรายชื่อคนไข้และบันทึกการตรวจ

---

## 2. เทคโนโลยีที่ใช้

| ส่วน | เทคโนโลยี |
|------|------------|
| **Frontend** | React 18, TypeScript, Vite 7, React Router 7, Tailwind CSS 4, Radix UI, Lucide Icons, Socket.io Client, SweetAlert2 |
| **Backend** | Node.js, Express 5, Mongoose (MongoDB ODM) |
| **ฐานข้อมูล** | MongoDB |
| **Real-time** | Socket.io (อัปเดตคิว การแจ้งเตือน) |
| **Authentication** | JWT (JSON Web Token), bcrypt สำหรับ hash รหัสผ่าน |

---

├── frontend/                # เว็บแอป (React + Vite)
│   ├── src/
│   │   ├── api/             # เรียก backend (auth, bookings, queues, ...)

## 4. บทบาทผู้ใช้ (Roles)

| Role | คำอธิบาย | หน้าที่หลัก |
|------|----------|-------------|
| **Patient (ผู้ป่วย)** | คนที่มาพบแพทย์ | สมัครสมาชิก, จองคิว, ดูใบจอง, เช็คอิน, ดูการแจ้งเตือน |
| **Admin (แอดมิน)** | เจ้าหน้าที่/พนักงานโรงพยาบาล | ล็อกอินแอดมิน, จัดการคิว (เรียก/ข้าม/จบ), ดูรายการจอง/เช็คอิน, ตั้งค่าแผนก-หมอ-ระบบ |
| **Doctor (แพทย์)** | หมอที่รับคนไข้ | ล็อกอินหมอ, ดูรายชื่อคนไข้, เรียกคนไข้เข้าห้อง, บันทึกหมายเหตุ, จบการตรวจ |

## 5. ฟีเจอร์หลักของระบบ

### ฝั่งผู้ป่วย (Patient)
- ลงทะเบียน / เข้าสู่ระบบ
- เลือกวันที่และแผนก → เลือกหมอ → เลือกช่วงเวลา (เช้า/บ่าย) → ยืนยันจอง
- ดูใบจอง (หมายเลขคิว, กำลังรับบริการ, เวลารอโดยประมาณ)
- เช็คอิน: ใส่รหัส/หมายเลขคิว หรือสแกน QR (แบบจำลอง) แล้วยืนยัน
- แดชบอร์ด: รายการนัดของตัวเอง, สถานะ, เวลารอโดยประมาณ
- การแจ้งเตือน: อ่าน/ลบ
- ตั้งค่าโปรไฟล์และเปลี่ยนรหัสผ่าน

### ฝั่งแอดมิน (Admin)
- ล็อกอินด้วยบัญชีแอดมิน
- แดชบอร์ด: สถิติวันนี้ (จำนวนจอง, เช็คอินแล้ว, รอ, กำลังตรวจ, เสร็จ), เทรนด์เทียบวันก่อน, ตารางคิวเช้า/บ่าย
- จัดการคิว: เรียกคิว, ข้ามคิว, จบการรับบริการ, เปลี่ยนสถานะ
- รายการรอ (Waiting List), รายการจอง (Bookings), รายละเอียดจอง (แก้ไข, ยกเลิก, เช็คอินมือ)
- Check-Ins: รายการเช็คอินตามวันที่, ปุ่มเช็คอินมือ
- Completed: รายการที่เสร็จแล้ว
- ตั้งค่า: ระบบ (ชื่อโรงพยาบาล, จำนวนคิว, เวลาทำการ), บัญชี, การแจ้งเตือน, แผนก, หมอ

### ฝั่งแพทย์ (Doctor)
- ล็อกอินด้วยบัญชีหมอ (ผูกกับโปรไฟล์หมอในระบบ)
- ดูรายชื่อคนไข้วันนี้ (รอ, กำลังตรวจ, เสร็จแล้ว)
- เรียกคนไข้เข้าห้อง (เปลี่ยนสถานะเป็น in-progress)
- เพิ่ม/แก้ไขหมายเหตุทางการแพทย์ (บันทึกลงระบบ)
- จบการตรวจ (completed) — ระบบอาจเลื่อนคิวถัดไปอัตโนมัติ

## 6. โมเดลข้อมูลหลัก (MongoDB)

- **User** — บัญชีผู้ใช้ (ผู้ป่วย, แอดมิน, หมอ): อีเมล, รหัสผ่าน, ชื่อ, เบอร์, วันเกิด, เพศ, บทบาท, การตั้งค่าการแจ้งเตือน
- **Department** — แผนก (ชื่อ, ลำดับการแสดง)
- **Doctor** — หมอ (ชื่อ, แผนก, เวลาทำการ, จำนวนคิววันนี้, หมายเลขที่กำลังรับบริการ, ลิงก์ User ถ้าเป็นหมอที่ล็อกอินได้)
- **Booking** — การจองคิว (หมายเลขคิว, หมอ, วันที่, ช่วงเวลา, ผู้ป่วย, สถานะ, หมายเหตุหมอ ฯลฯ)
- **CheckIn** — บันทึกการเช็คอิน (การจอง, ผู้ป่วย, วิธีเช็คอิน, สถานะ)
- **Notification** — การแจ้งเตือน (ผู้ใช้, หัวข้อ, ข้อความ, อ่านแล้วหรือยัง)
- **SiteConfig** — ตั้งค่าระบบ (เอกสารเดียว): ชื่อโรงพยาบาล, จำนวนคิวต่อช่วง/ต่อวัน, เวลาทำการ

---

## 7. ความปลอดภัยและสิทธิ์ (สรุป)

- **Authentication:** JWT เก็บใน localStorage ฝั่ง frontend ส่งใน header `Authorization: Bearer <token>`
- **Authorization:** Middleware ตรวจ role (admin, doctor) สำหรับ API ที่อ่อนไหว
- **Admin API** (จัดการคิว, settings, checkins ฯลฯ): ต้องล็อกอินและเป็น role **admin**
- **Doctor API** (สร้าง/แก้/ลบหมอ): เฉพาะ **admin**; เรียกคิว/บันทึกหมายเหตุ: **admin** หรือ **doctor**
- **Department API** (สร้าง/แก้/ลบแผนก): เฉพาะ **admin**
- **Booking API:** สร้าง/ดู/แก้/ยกเลิก ต้อง **ล็อกอิน**; ดูรายการได้เฉพาะของตัวเอง หรือแอดมินดูทั้งหมดได้
