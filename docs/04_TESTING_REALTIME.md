# วิธีทดสอบการอัปเดตแบบเรียลไทม์ (Realtime)

เอกสารนี้อธิบายวิธีทดสอบว่าข้อมูลบนเว็บอัปเดตทันทีโดยไม่ต้องกด Refresh เมื่อมีการเรียกคิว / เปลี่ยนสถานะ / เช็คอิน

---

## สิ่งที่ต้องมีก่อนทดสอบ

1. **Backend รันอยู่** — ในโฟลเดอร์ `backend` รัน `npm run dev` (พอร์ต 3001)
2. **Frontend รันอยู่** — ในโฟลเดอร์ `frontend` รัน `npm run dev` (พอร์ต 5173)
3. **มีข้อมูล seed แล้ว** — รัน `node seed.js` ในโฟลเดอร์ backend อย่างน้อยครั้งหนึ่ง

## วิธีทดสอบ: เปิดหลายแท็บ

### แท็บที่ 1: ผู้ป่วย

1. เปิดเบราว์เซอร์ไปที่ `http://localhost:5173` (หรือ URL ของ frontend)
2. ล็อกอินด้วย **ผู้ป่วย** (เช่น `somchai.p@email.com` / `password123`)
3. ไปที่ **Dashboard** (`/dashboard`) — อยู่ที่หน้านี้ไว้ **ไม่ต้องกด Refresh**

### แท็บที่ 2: แอดมิน หรือ หมอ

4. เปิด **แท็บใหม่** (หรือหน้าต่างใหม่) ไปที่ `http://localhost:5173`
5. ล็อกอินด้วย **แอดมิน** (`admin@mediqueue.com` / `password123`) หรือ **หมอ** (`doctor1@mediqueue.com` / `password123`)
6. ไปที่หน้า:
   - **แอดมิน:** `/admin` → **Queue** (หรือ Dashboard ที่มีตารางคิว)
   - **หมอ:** `/doctor`

### ทดสอบการเรียกคิว (Call) — ฝั่งผู้ป่วยได้ป๊อปอัพ + ข้อมูลอัปเดต

7. ที่แท็บ **แอดมิน/หมอ** หารายการคิวของ **ผู้ป่วยที่ล็อกอินอยู่ในแท็บ 1** (เช่น สมชาย ใจดี / somchai.p@email.com)
8. กด **Call Next** (แอดมิน) หรือ **Call** (หมอ) สำหรับคิวนั้น
9. **สลับไปที่แท็บผู้ป่วย** (ไม่ต้องกด Refresh):
   - ควรเห็น **ป๊อปอัพแจ้งเตือน** มุมขวาบน เช่น "ถึงคิวของคุณแล้ว!" พร้อมหมายเลขคิว แพทย์ แผนก
   - หน้า Dashboard ควรแสดงสถานะการจองเป็น **In Progress** (หรืออัปเดตตามสถานะล่าสุด)

### ทดสอบแอดมิน/หมอ — ข้อมูลอัปเดตโดยไม่ Refresh

10. อยู่ที่แท็บ **แอดมิน** หน้า Queue (หรือ Dashboard):
    - ให้อีกคน (หรือเปิดแท็บที่ 3 ล็อกอินเป็นหมอ) กด **Call** หรือ **Complete** ที่คิวใดก็ได้
    - หน้าแอดมินควร **อัปเดตรายการคิวทันที** (สถานะเปลี่ยน เช่น เป็น In Progress / Completed) โดยไม่กด Refresh

11. อยู่ที่แท็บ **หมอ** หน้า `/doctor`:
    - ให้แอดมินกด **Call Next** สำหรับคิวของหมอคนนี้
    - หน้าหมอควร **อัปเดตรายการคิวทันที** (คนที่ถูกเรียกย้ายจาก Waiting ไป In Consultation) โดยไม่กด Refresh

### ทดสอบ Check-Ins เรียลไทม์

12. แท็บแอดมิน ไปที่ **Check-Ins** (`/admin/checkins`)
13. แท็บอีกอัน: ผู้ป่วยล็อกอินแล้วไปหน้า **Check-in** (`/check-in`) แล้วเช็คอินด้วยหมายเลขคิวหรือรหัส
14. หน้าแอดมิน **Check-Ins** ควร **อัปเดตรายการเช็คอินทันที** โดยไม่กด Refresh

## สรุปสิ่งที่ควรเห็น (เรียลไทม์)

| ฝั่งที่ดู | เหตุการณ์ | สิ่งที่ควรเห็นโดยไม่ Refresh |
|-----------|-----------|-------------------------------|
| **ผู้ป่วย** | หมอ/แอดมินเรียกคิว | ป๊อปอัพแจ้งเตือน + Dashboard อัปเดตสถานะ |
| **ผู้ป่วย** | หมอจบการตรวจ (Complete) | Dashboard อัปเดตสถานะเป็น Completed |
| **แอดมิน** | เรียกคิว / ข้าม / จบ / เช็คอิน | หน้ารายการคิว / Dashboard / Check-Ins อัปเดตทันที |
| **หมอ** | แอดมินเรียกคิว หรือมีการอัปเดต booking | หน้ารายการคิวของหมออัปเดตทันที |

## ข้ามคิวอัตโนมัติ (10 นาที)

ระบบมี **การข้ามคิวอัตโนมัติ** ตามข้อความในป๊อปอัพ:

- เมื่อหมอ/แอดมิน **เรียกคิว** (Call) จะบันทึกเวลาที่เรียก (`calledAt`)
- Backend มี job รันทุก **1 นาที** ตรวจสอบรายการที่สถานะ **in-progress** และถูกเรียกมาแล้ว **เกิน 10 นาที**
- ถ้าผู้ป่วยยังไม่มาหรือยังไม่จบการตรวจ ระบบจะ **ข้ามคิวอัตโนมัติ** (สถานะ → canceled) และ **เรียกคิวถัดไป** ให้
- ผู้ป่วยที่ถูกข้ามจะได้ **การแจ้งเตือน** "Queue Skipped - no response within 10 minutes"
- คิวถัดไปจะได้การแจ้งเตือน "Your Queue is Called!" แบบเรียลไทม์

**วิธีทดสอบ (ต้องรอหรือปรับเวลา):** เรียกคิวผู้ป่วยหนึ่งคน แล้วไม่กด Complete/Skip — รอเกิน 10 นาที (หรือลดค่า `MINUTES_BEFORE_AUTO_SKIP` ใน `backend/jobs/autoSkipQueue.js` ชั่วคราวเป็น 1 นาที เพื่อทดสอบ) จากนั้นดูที่แอดมิน/หมอว่าคิวถูกข้ามและเรียกคิวถัดไปหรือไม่ และผู้ป่วยที่ถูกข้ามได้แจ้งเตือนหรือไม่

## ถ้าอัปเดตไม่ทันที

1. **ตรวจว่า Backend รันอยู่** — Socket ใช้พอร์ตเดียวกับ API (เช่น 3001)
2. **ตรวจ Console ใน DevTools (F12)** — ดูว่ามี error เกี่ยวกับ WebSocket หรือ socket.io หรือไม่
3. **ลอง Refresh หน้าแล้วทดสอบอีกครั้ง** — เพื่อให้เชื่อม Socket ใหม่
4. **ตรวจว่าใช้ URL เดียวกัน** — ฝั่ง frontend ต้องชี้ไปที่ backend เดียว (เช่น `VITE_API_URL=http://localhost:3001`)
